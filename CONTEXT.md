## 1. What is Crucible?

Crucible is a from-scratch ONNX inference engine written in C++17.
It does the same job as ONNX Runtime and TensorFlow Lite — load a `.onnx`
model, parse the compute graph, run inference on CPU — but every layer is
written from scratch: the tensor class, the ONNX protobuf parser, each
operator, and the graph executor.

The project spans five languages:

| Language   | Role                                               |
|------------|----------------------------------------------------|
| C++17      | Core engine — tensors, operators, graph executor   |
| Python     | pybind11 bindings + FastAPI inference server       |
| Rust       | CLI tool (FFI into C++ engine) + WASM module       |
| TypeScript | Next.js 15 web demo (WASM in browser)              |
| CMake      | Build system for the C++ engine                    |

## 4. Build Commands

```bash
# C++ engine — configure
cmake --preset debug -S engine
cmake --preset release -S engine

# C++ engine — build
cmake --build engine/build/debug --parallel
cmake --build engine/build/release --parallel

# C++ engine — test
cd engine/build/debug && ctest --output-on-failure

# Python bindings — install after building
pip install -e engine/bindings/python

# FastAPI server
cd server && uvicorn app.main:app --reload --port 8000

# Rust CLI
cd cli && cargo build --release

# WASM build (requires wasm-pack installed)
bash scripts/build-wasm.sh

# Next.js web
cd web && npm install && npm run dev
```

## 5. Key Conventions

### Commit message format
```
feat(scope): short description
fix(scope): short description
test(scope): short description
chore(scope): short description
ci: short description
docs: short description
```
Valid scopes: `engine`, `ops`, `executor`, `bindings`, `server`, `cli`,
`wasm`, `web`, `bench`, `models`, `scripts`.

### C++ style rules
- All operator implementations live under `engine/src/ops/` and `engine/include/crucible/ops/`
- Every operator signature: `Tensor <name>_forward(const Tensor& input, ...)`
- Use Eigen MatrixXf for all linear algebra (already a submodule in engine/)
- Numerical tolerances in tests: `EXPECT_NEAR(..., 1e-5)` for float ops, `1e-4` for conv
- No raw pointers inside engine core — use `std::vector<float>` for tensor data

### Python style rules
- FastAPI with Pydantic v2 only (no v1 compat)
- All pytest files in `server/tests/`, named `test_*.py`
- Model files stored in `CRUCIBLE_MODEL_DIR` env var (default: `./models/`)

### Rust style rules
- Rust edition 2021
- `cargo clippy -- -D warnings` must pass before any commit
- FFI functions in `engine/src/c_api.cpp` use `extern "C"` with no C++ exceptions crossing the boundary

### TypeScript / Next.js rules
- Next.js 15 App Router only — no Pages Router
- Server components by default; add `"use client"` only where needed
- WASM loaded via dynamic import with lazy init in `crucible-wasm.ts`
- No `<form>` tags — use `onClick`/`onChange` handlers

## 6. Critical Architecture Notes

### WASM is pure Rust — C++ does NOT compile to WASM
The `wasm/` module is a standalone pure-Rust reimplementation of a subset
of ops (MatMul, ReLU, Softmax, Sigmoid). The C++ engine cannot be compiled
to WebAssembly. Never attempt to compile `engine/` with wasm-pack.

### Tensor memory layout
Row-major (C-order). Shape `{N, C, H, W}` — NCHW, same as PyTorch default.
Data stored as `std::vector<float>`. Index: `i*C*H*W + c*H*W + h*W + w`.

### ONNX model weights are initializers
When parsing `.onnx` files, model weights come from `graph.initializer[]`,
not from `graph.input[]`. The executor pre-loads all initializers into
`tensor_map` before running the graph.

### Protobuf dependency
The ONNX protobuf parsing uses the `onnx.proto3` schema compiled with
`protoc`. The generated headers live in `engine/include/onnx/`. Do not
regenerate unless the ONNX opset version changes (currently opset 7).

### MobileNetV2 ops used
Clip (ReLU6), Conv, BatchNormalization, Add, GlobalAveragePool, Flatten,
Gemm. Any op not in this list is not needed for MobileNetV2 end-to-end.

### Benchmark Data (Measured on 2026-07-04)

## 10. Known Issues & Gotchas

1. **CMake preset names are lowercase**: always `debug` and `release`, never `Debug` or `Release`.
2. **Eigen is a submodule** at `engine/third_party/eigen`. If it's empty, run `git submodule update --init --recursive`.
3. **pybind11 zero-copy**: the numpy ↔ Tensor bridge uses `py::array_t<float, py::array::c_style | py::array::forcecast>` — the `forcecast` ensures non-contiguous arrays are handled.
4. **WASM has no C++ engine**: the `wasm/` module reimplements MatMul, ReLU, Softmax, Sigmoid in pure Rust. It does not call `libcrucible`.
5. **MobileNetV2 opset**: we use opset 7 (mobilenet-v2-7.onnx). Don't swap in a newer opset file without verifying the executor supports the new ops.
6. **Rust FFI safety**: `c_api.cpp` catches all C++ exceptions internally and returns error codes. The Rust `runner.rs` checks return codes — never assume a non-null pointer means success.
7. **Next.js WASM config**: `asyncWebAssembly: true` and `layers: true` must both be set in `next.config.ts`. Missing either causes a runtime module-not-found error.
8. **benchmark_results.json is gitignored**: it's generated by `benchmarks/run_all.py` on the local machine. The web benchmark page has a local JSON fallback import for CI.
9. **Depthwise/Grouped Convolution support**: General Conv2D grouped convolution (groups > 1) is supported, enabling full compatibility with MobileNetV2 depthwise separable convolution (where groups == in_channels). It validates that groups divide input/output channels evenly, and runs im2col + GEMM independently per group.
